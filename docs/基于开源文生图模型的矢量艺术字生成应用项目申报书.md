## 二、项目背景与立项意义

艺术字是品牌标识、营销海报、活动视觉、商品包装、UI 素材和新媒体内容中高频使用的设计元素。传统艺术字制作主要依赖设计师在 Photoshop、Illustrator、Inkscape 等工具中手工绘制，具有创作周期长、批量生成难、风格复用成本高、交付结果难以自动化验收等问题。随着开源文生图模型和本地图像处理工具链的发展，使用自然语言生成高质量艺术字位图已经具备可行性，但直接生成的位图仍然存在两个关键限制：一是文字准确性、主体完整性和透明背景稳定性需要工程化约束；二是位图不具备专业设计工作流所要求的可编辑、可缩放和可拆分能力。

本项目围绕赛题“基于开源文生图模型的矢量艺术字生成应用”的要求，构建一套本地优先的艺术字生成与矢量化系统。系统以 ComfyUI 和开源文生图模型作为图像生成基础，通过提示词模板、工作流降级链和结构化元数据提升生成可控性；以 Python 图像处理流水线完成背景移除、边缘降噪、颜色量化、路径追踪和 SVG 回渲染质量评估；以 Electron + Vue 3 桌面端和 Node.js CLI 提供可视化操作与批量验收能力。项目完成后，用户可以输入目标文字与风格描述，一键生成艺术字图片，并进一步转换为可在主流矢量设计软件中编辑的标准 SVG 文件。

本项目的立项意义体现在以下方面：

1. 提升艺术字生产效率。将“文字创意到可编辑矢量素材”的流程自动化，降低非专业用户制作艺术字素材的门槛。
2. 推动 AIGC 与传统设计软件衔接。生成结果不仅是可预览图片，而且包含真实 SVG 路径，便于后续二次编辑和商业设计流程使用。
3. 强化本地化与数据安全。核心推理、图像处理、矢量化和产物管理均默认在本地完成，避免将提示词和图片上传至闭源第三方服务。
4. 形成可复现、可验收的工程系统。项目提供标准化输出目录、元数据、日志、批量测试集和验收脚本，便于评审和后续迭代。

## 三、赛题要求理解与需求分析

根据赛题文档，项目应实现从文本提示词到矢量艺术字的完整闭环，核心要求包括：

1. 基于开源文生图模型搭建可本地运行的 ComfyUI 工作流，不依赖闭源商业 API 完成核心生成任务。
2. 支持中文、英文、数字和符号等多类型艺术字生成，并通过提示词控制字体风格、颜色、材质、装饰元素、版式结构和背景透明效果。
3. 对生成位图进行预处理，包括背景分离、边缘去噪、主体裁剪、颜色量化和抗锯齿保留。
4. 使用 Python 程序将位图转换为标准 SVG，要求包含真实可编辑的路径或形状元素，不得以 Base64 位图嵌入伪装为 SVG。
5. 输出原始位图、透明 PNG、SVG、SVG 回渲染预览图、JSON 元数据和运行日志。
6. 支持单条生成、批量生成和已有图片矢量化三类使用模式。
7. 具备可复现的参数记录、模型依赖说明、异常日志和自动化验收能力。

## 四、项目建设目标

### 4.1 总体目标

建设一款面向 Windows 本地环境的矢量艺术字生成应用。用户输入艺术字文本、风格提示词、负面提示词、分辨率、随机种子和矢量化参数后，系统自动调用开源文生图工作流生成艺术字位图，再通过本地图像处理和矢量化流水线输出透明 PNG 与标准 SVG，形成可编辑、可缩放、可复现、可批量生成的艺术字素材生产工具。

### 4.2 功能目标

1. 实现文本到艺术字位图生成。支持中文、英文、数字、混合文本和促销信息等常见艺术字内容。
2. 实现多风格控制。支持国风品牌、海报促销、节日活动、英文标题、混合文本等风格场景。
3. 实现透明背景处理。对已有 alpha 通道图片保留透明信息，对普通图片使用本地 rembg 模型进行背景移除。
4. 实现标准 SVG 矢量化。通过颜色分层和路径追踪生成可编辑的 `<path>`、`<g>` 等矢量结构。
5. 实现质量指标记录。计算 PNG 透明度、SVG 回渲染还原度、耗时、文件大小等指标。
6. 实现多入口使用。桌面端面向普通用户，CLI 面向评审验收、批处理和自动化脚本。
7. 实现批量任务管理。支持 TXT、CSV、JSON 输入，逐条容错执行，生成 `batch_summary.csv` 汇总。
8. 实现标准化产物管理。每个任务输出 `original.png`、`transparent.png`、`result.svg`、`preview.png`、`metadata.json`、`run.log` 和工作流快照。

### 4.3 性能与可靠性目标

1. 单张 1024 级别图像矢量化处理在常规硬件上控制在可交互范围内，优先满足自动化批量验收稳定性。
2. 批量处理不少于 100 条提示词，项目内置 300 条验收集用于覆盖多类场景。
3. 文生图工作流失败时自动尝试候选工作流；ComfyUI 不可用时降级到 Pillow 本地 stub，保证接口返回可诊断结果。
4. 输出 SVG 必须可被浏览器渲染，且验收脚本检查不包含 `<image>` 或 Base64 位图伪 SVG。
5. 所有异常写入日志或汇总表，避免批量任务因单条失败整体中断。

### 5.1 文生图服务

`txt2img-api` 使用 FastAPI 提供 `/api/v1/txt2img` 接口。服务负责加载 ComfyUI API 格式工作流 JSON，按请求注入正向提示词、负向提示词、分辨率和随机种子，然后通过 ComfyUI 的 `/prompt`、`/history/{prompt_id}`、`/view` 接口完成异步生成和图片获取。

该服务具备以下工程特性：

1. 工作流按模型类型自动构造提示词，增强中文、英文和数字内容的准确性约束。
2. 根据输入文本自动选择工作流降级链：中文优先使用 Qwen-Image 工作流，英文或其他文本优先使用 Flux 工作流，失败后切换 Z-Image 工作流。
3. 对 Qwen-Image 官方分辨率进行映射，兼容 1:1、16:9、9:16、4:3、3:4、3:2、2:3 等常见比例。
4. 生成结果包含 `workflow_used`、`fallback_tier`、`attempted_workflows`、模型依赖快照和工作流快照，便于复现与审查。
5. 当 ComfyUI 不可达或工作流失败时，降级到本地 Pillow stub，保证桌面端和 CLI 能得到可诊断响应。

### 5.2 矢量化服务

`vectorizer-api` 使用 FastAPI 提供 `/api/v1/vectorize` 接口。服务可接收用户上传图片，也可接收文生图阶段生成的图片路径或 Base64 数据。核心处理流程为：

1. 解码输入图片，识别是否已有 alpha 通道。
2. 无 alpha 时调用本地 `isnet-general-use.onnx` rembg 模型移除背景。
3. 使用 OpenCV 对透明边缘进行去噪、形态学修复、小连通域过滤和颜色清理。
4. 使用 vtracer 进行颜色分层路径追踪，输出 SVG。
5. 使用 CairoSVG 将 SVG 回渲染为 PNG 预览。
6. 使用 SSIM、梯度相关性和前景色分布计算 SVG 还原度。
7. 输出透明 PNG、SVG、预览图和结构化元数据。

服务内置四组矢量化预设：

| 预设 | 适用场景 | 特点 |
| --- | --- | --- |
| `clean` | 干净图形、少色块素材 | 路径更少，文件更小 |
| `balanced` | 默认批量处理 | 平衡细节与体积 |
| `detailed` | 细节丰富的艺术字 | 保留更多边缘和颜色层次 |
| `ultra` | 高保真样例 | 最大细节，文件体积较大 |

### 5.3 桌面端应用

桌面端位于 `apps/desktop`，采用 Electron + Vue 3 + Vite 实现。桌面端承担普通用户操作入口、任务编排、结果预览、历史记录和打包后后端进程管理职责。界面支持三类模式：

1. 单条生成：输入文本和风格提示词，生成一组完整产物。
2. 批量生成：导入或粘贴多条提示词，按顺序生成并输出汇总表。
3. 图片矢量化：选择本地 PNG/JPG 图片，跳过文生图阶段直接输出 SVG。

### 5.4 CLI 自动化工具

CLI 位于 `apps/cli`，定位为随桌面端安装包交付的自动化验收控制台。它不负责后端生命周期管理，而是通过 HTTP 调用已启动的 `txt2img-api` 和 `vectorizer-api`。主要命令包括：

| 命令 | 作用 |
| --- | --- |
| `pipeline` | 完整流水线：文本生成位图，再矢量化为 SVG |
| `generate` | 仅执行文生图生成 |
| `vectorize` | 仅对已有图片做矢量化 |
| `batch` | 批量执行流水线 |
| `health` | 检查后端健康状态 |
| `env` | 输出运行环境和服务地址 |
| `shutdown` | 关闭后端服务 |

## 六、关键技术路线

### 6.1 开源文生图模型与 ComfyUI 工作流

项目使用 ComfyUI 作为开源文生图工作流编排与推理平台。工作流文件统一保存在 `services/txt2img-api/workflows/`，采用 ComfyUI API 格式 JSON，便于后端自动加载、修改和提交。

技术实现要点包括：

1. 通过 `class_type` 扫描方式识别 `CLIPTextEncode`、`CLIPTextEncodeFlux`、`EmptyLatentImage`、`KSampler`、`SaveImage` 等关键节点，避免强依赖固定节点编号。
2. 对正向提示词加入背景抑制、文字准确性、字符数量、笔画完整性等约束，提高艺术字主体可读性。
3. 对负面提示词合并错字、漏字、断笔、复杂背景、水印、低清晰度等默认排除项。
4. 输出工作流快照和模型依赖清单，满足赛题对可复现交付的要求。

### 6.2 提示词结构化与文字准确性约束

艺术字生成的主要难点是模型容易出现错字、漏字、重复字符、笔画粘连和背景干扰。项目在提示词构造中引入如下策略：

1. 中文文本拆分为单字列表，并明确字符数量。
2. 英文文本强调 crisp typography、perfect letterforms、no duplicate letters。
3. 数字内容加入 accurate digits、no repeated digits 等约束。
4. 默认注入纯色、简洁、无复杂场景背景要求，便于后续透明背景处理。
5. 用户负面提示词与系统负面提示词合并，保留用户控制能力。

### 6.3 透明背景与图像预处理

矢量化前的图像质量直接决定 SVG 可编辑性与轮廓稳定性。项目使用以下预处理策略：

1. 对已有 alpha 通道的透明 PNG，保留原透明信息，避免重复抠图导致边缘损伤。
2. 对普通 PNG/JPG，使用本地 rembg `isnet-general-use.onnx` 模型进行背景移除。
3. 对 alpha 通道执行阈值处理、小连通域过滤、形态学闭运算/开运算和边缘平滑。
4. 对 RGB 区域执行双边滤波，降低噪点同时保留主体边缘。
5. 根据预设对 near-white 边缘进行清理，减少透明背景中的白边残留。

### 6.4 SVG 路径追踪与质量评估

项目使用 vtracer 执行颜色分层路径追踪，选择 `colormode=color`、`hierarchical=stacked`、`mode=spline` 等模式输出真实 SVG 矢量路径。矢量化参数可配置颜色精度、小噪点过滤、角点阈值、路径片段长度阈值、图层差异和缩放倍率。

为避免只输出 SVG 而无法判断质量，项目引入回渲染闭环：

1. 使用 CairoSVG 将 SVG 回渲染为 PNG。
2. 将透明区域统一填充为中性灰，避免透明背景干扰评分。
3. 对原图与回渲染图计算 SSIM。
4. 使用 Sobel 梯度相关性评估边缘形态一致性。
5. 使用 Lab 色彩空间前景色分布评估主体颜色还原。
6. 汇总得到 `svg_fidelity` 指标并写入 `metadata.json` 与批量汇总表。

### 6.5 标准化产物与可复现机制

每个任务输出独立目录，默认产物如下：

```text
task_YYYYMMDD_HHMMSS_xxx/
├─ original.png
├─ transparent.png
├─ result.svg
├─ preview.png
├─ metadata.json
├─ run.log
└─ workflows/
   ├─ workflow_api.json
   ├─ nodes.md
   └─ model_dependencies.json
```

其中：

1. `metadata.json` 记录文本、提示词、负面提示词、分辨率、随机种子、矢量化参数、质量指标、耗时和文件路径。
2. `run.log` 使用 key=value 形式记录关键运行状态，便于脚本解析。
3. `workflow_api.json` 保存实际提交给 ComfyUI 的工作流，保证参数复现。
4. `model_dependencies.json` 保存运行时扫描到的模型依赖，方便审查模型来源。

## 七、项目创新点

1. 文生图与矢量化闭环结合。项目不止生成艺术字位图，而是进一步完成透明背景、颜色分层、路径追踪和 SVG 回渲染评价，使结果能进入专业设计编辑流程。
2. 本地优先的 AIGC 设计工具形态。系统默认在本地运行 ComfyUI、rembg 和矢量化算法，兼顾数据安全、可复现和离线可用性。
3. 面向艺术字的提示词增强策略。系统根据中文、英文、数字混合内容动态构造提示词，重点约束字符数量、笔画完整、错漏字和背景复杂度。
4. 工作流降级链与可诊断兜底。不同模型工作流失败时自动切换候选工作流；ComfyUI 不可用时仍提供本地 stub 响应，便于界面和验收流程继续运行。
5. SVG 质量闭环评价。通过回渲染 PNG 与透明 PNG 的视觉相似度指标，给出可量化的 SVG 还原度，而不是只依赖人工观察。
6. 桌面端与 CLI 双入口共用后端能力。普通用户使用 GUI，评审和开发使用 CLI 批量验收，减少重复实现。
7. 自动化验收脚本覆盖交付要求。PowerShell 验收脚本会检查批量汇总、产物完整性、PNG 合法性、透明通道、SVG XML 结构、禁止伪 SVG、元数据和日志字段。

| 自动化测试 | pytest、PowerShell 验收脚本 |

### 8.3 测试基础

项目提供 `tests/fixtures/acceptance.txt` 作为标准批量验收集，包含 300 条提示词，覆盖国风品牌、海报促销、节日活动、英文标题、混合文本等类别。`tests/run-acceptance.ps1` 支持以下检查：

1. 批量汇总 CSV 行数和失败条数。
2. 每个任务目录是否包含标准产物。
3. PNG 文件是否可读、尺寸是否符合预期或记录为可解释差异。
4. `transparent.png` 是否具备 alpha 通道。
5. `result.svg` 是否为合法 XML，是否有 SVG 根元素，是否包含可编辑矢量元素。
6. 禁止 SVG 中包含 `<image>` 或 Base64 位图数据。
7. 元数据 JSON 是否合法，日志是否包含状态与耗时字段。
8. 可选执行端到端耗时和矢量化耗时阈值检查。

## 十三、风险分析与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 开源文生图模型文字准确率不稳定 | 可能出现错字、漏字、重复字 | 提示词中强化字符数量和笔画完整约束；中文优先使用 Qwen-Image 工作流；保留 seed 与工作流用于复现 |
| ComfyUI 工作流或模型缺失 | 文生图服务无法完成推理 | 提供模型依赖清单、下载脚本、健康检查和本地 stub 兜底 |
| 背景移除边缘有白边或噪点 | 影响矢量路径质量 | 使用 rembg + OpenCV 边缘清理、小连通域过滤、预设化参数调节 |
| SVG 路径过碎或文件过大 | 影响编辑体验和性能 | 提供 clean、balanced、detailed、ultra 四级预设，按场景权衡细节和体积 |
| 批量任务单条失败中断整体流程 | 影响验收稳定性 | CLI 逐条容错，写入失败状态和错误信息，汇总 CSV 便于追踪 |
| Windows 打包资源不完整 | 评审环境开箱失败 | 使用 `electron-builder` extraResources 明确打包后端、CLI、测试集、模型目录和脚本 |
| 第三方模型许可证不一致 | 影响分发与商用 | 在模型依赖文档中明确来源、版本和许可，源码 MIT 与模型许可分开说明 |

## 十七、结论

Gen2Vec ArtFont 项目围绕“开源文生图模型生成艺术字”和“位图高保真矢量化”两项核心任务，构建了从提示词输入、ComfyUI 生成、透明背景处理、SVG 路径追踪、质量评估、标准产物输出到批量验收的完整工程闭环。项目既重视 AIGC 生成效果，也重视矢量设计交付中更关键的可编辑性、可复现性和可验收性。

结合当前代码基础，项目已经具备桌面端、CLI、文生图后端、矢量化后端、验收测试集和打包交付框架，后续重点将集中在模型资源完善、生成质量优化、示例产物打磨和最终交付文档完善。项目技术路线清晰、工程基础完整、应用价值明确，符合赛题关于本地部署、开源模型、Python 矢量化、标准 SVG 输出和自动化验收的综合要求，具备较高的实施可行性和展示价值。
